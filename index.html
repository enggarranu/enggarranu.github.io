Engineering Principles

How I make migration decisions.

The same principles apply whether the cutover moves 4TB or 40TB.

01

Build Platforms, Not Projects

The most valuable engineering work creates reusable systems rather than isolated, one-off migrations.

02

Design for Rollback First

Every production cutover ships with a tested recovery path before it ships with a deadline.

03

Automation Reduces Operational Risk

Manual, repetitive steps are where migrations fail — automate provisioning, validation, and rollback out of human hands.

04

Reliability Before Velocity

Moving fast has no value if a 40TB cutover isn't safe to reverse.

05

Standardization Enables Scale

408+ clusters can't run on individual tribal knowledge — they run on shared, documented standards.

06

Measure Before Optimizing

Validation, checksums, and dashboards come before any performance or reliability claim.

07

Documentation Is Part of Engineering

A migration isn't finished until another engineer can run it without me in the room.

08

Own the Outcome

Ownership extends past cutover — into monitoring, incident response, and the next migration wave.

PostgreSQL → MySQL Migration Platform

Tech Lead for Tokopedia's PostgreSQL-to-MySQL migration program following the ByteDance acquisition — one of the company's largest database modernization initiatives.

Migration Architecture

Problem

  • Production ran on PostgreSQL while the post-acquisition platform standardized on MySQL, at a scale where dump-and-restore wasn't viable.

Engineering Decisions

  • Co-designed CDC-based migration architecture (Debezium → Kafka) instead of downtime-heavy bulk copies.
  • Separated validation from execution and designed rollback before cutover.
  • Served as a technical approver in Go/No-Go production decisions.

Architecture

  • Provisioning → CDC replication → validation → cutover → rollback, all standardized as reusable platform capabilities.

Migration Architecture

The lifecycle behind every safe cutover.

Every migration — from a single service to the full PSM program — moves through the same eight phases.

01

Discovery

Analyze application architecture, production risk, and migration complexity.

02

Architecture Design

Select migration strategy, define topology, design rollback, plan validation.

03

Infrastructure Provisioning

Provision migration infrastructure and configure CDC environments.

04

Migration Execution

Initial snapshot, CDC synchronization, and continuous data verification.

05

Production Validation

Schema, data, application, and performance validation before cutover.

06

Cutover

Switch production traffic and verify platform health in real time.

07

Rollback

Recover safely if required — protecting business continuity above all.

08

Post-Migration

Monitor, review operationally, and capture lessons learned.
